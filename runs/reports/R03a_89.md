## Полная матрица 2×2 — арх × MPS

| Тест | sm_120 MPS OFF | sm_120 MPS ON | sm_89 MPS OFF | sm_89 MPS ON |
|---|---|---|---|---|
| **1** ctx identity | gotorch=`0x0b14d730` goml=`0x0cbd6c70` **разные** | gotorch=`0x2f0e1730` goml=`0x30b40a50` **разные** | gotorch=`0x06e94910` goml=`0x09000070` **разные** | gotorch=`0x18d15910` goml=`0x1a2be0b0` **разные** |
| **2** Entry door | 1024/1024 bit-exact | 1024/1024 | 1024/1024 | 1024/1024 |
| **3** Exit door | 1024/1024 | 1024/1024 | 1024/1024 | 1024/1024 |
| **4** Cross-teardown | 128/128 | 128/128 | 128/128 | 128/128 |
| **5** ThreadMix 100 iter | 100/100+100/100 | 100/100+100/100 | 100/100+100/100 | 100/100+100/100 |
| **6A** ctx injection via API | недоступно | недоступно | недоступно | недоступно |
| **6B** retain primary → gotorch AddF32 | 1024/1024 (совпадает с snap) | 1024/1024 (совпадает) | 1024/1024 (совпадает) | 1024/1024 (совпадает) |
| **6C** UVA memcpy floating→primary | 1024/1024 | 1024/1024 | 1024/1024 | 1024/1024 |
| **6D** kernel launch cross-ctx | 1024/1024 | 1024/1024 | 1024/1024 | 1024/1024 |

**Все 4 клетки: 6/6 subtest PASS bit-exact.** Ни одна операция не даёт INVALID_CONTEXT/ILLEGAL_ADDRESS ни на одной конфигурации. **Вопрос UVA-контекстов закрыт полной матрицей.**

**Наблюдение по floating ctx handle (Test 6C):**

| Клетка | Floating ctx handle |
|---|---|
| sm_120 MPS OFF | `0x092af0f0` (низкие 32-bit) |
| sm_120 MPS ON | `0x31e51b70` (низкие 32-bit) |
| sm_89 MPS OFF | (не измерен отдельно; ctx handles Test 1 в низких 32-bit) |
| sm_89 MPS ON | `0x00007207e816d250` (полное 64-bit, high-half занят) |

**High-half-shift оказался спецификой sm_89+MPS driver 580.126.20**, а не универсальным поведением MPS. На sm_120+MPS handle остаётся в низких 32-bit. Причина скорее всего — разница в memory-map layout между Ada-driver (12.8-stack) и Blackwell-driver (13.1-stack). Полезная контр-точка: подтверждает, что «работает на sm_89» → «работает на sm_120» **автоматически не переносится** численно, но результат bit-exactness одинаков в обеих архитектурах.

## Ключевые находки на sm_89

### 1. PTX `.target sm_80` JIT-ится на sm_89 без проблем

goml PTX-модули содержат `.target sm_80` (forward-compat минимум), gotorch тоже. Инициализация обеих CUDA-backend'ов на sm_89 прошла без INVALID_PTX/INVALID_IMAGE:

```
[GoML] cublasGemmEx wrapper loaded -- FP16 mixed precision available
[GoML] cuBLASLt FP8 wrapper loaded -- FP8 E4M3 available
[GoML] CUDA backend initialized: NVIDIA GeForce RTX 4090 (Ada Lovelace, sm_89)
```

Заявление о forward-compat подтверждено. Ни один PTX-модуль не пришлось пересобирать.

### 2. Wrapper'ы FP16/FP8 — оба подключились

Обе wrapper библиотеки (`libcublas_wrapper.so`, `libcublaslt_wrapper.so`) собрались на месте на gpu4 из закоммиченных исходников (`libs1/cublas_wrapper.c` + `libs1/cublas_lt_wrapper_v3.c`) через `scripts/build_cuda.sh cublas_wrapper cublaslt_wrapper` + CUDA 12.8 headers. `sm_89` (compute 8.9) удовлетворяет требованию FP8 E4M3 в cublasLt (нужен ≥ 8.9 Hopper/Ada) — wrapper активировался с явной строкой `-- FP8 E4M3 available`.

### 3. MPS не влияет на interop — прямая A/B-проба

**Провели прогон дважды:**

1. **MPS OFF** — старый server (PID 145436) убит через `kill 145436` (стандартный `echo quit | nvidia-cuda-mps-control` не сработал — control pipe оказался изолирован после того как systemd/пользователь перезапускали daemon). После выключения нашего теста — MPS восстановлен новым daemon (PID 1725293 server + 1719177 control).
2. **MPS ON** — прогон с явными `CUDA_MPS_PIPE_DIRECTORY=/tmp/nvidia-mps` + `CUDA_MPS_LOG_DIRECTORY=/tmp/nvidia-log`, чтобы наш процесс подключился как MPS-client.

**Результат:** 6/6 PASS bit-exact **в обоих режимах**. Кросс-контекстная прозрачность НЕ зависит от MPS.

**Интерпретация:** UVA обеспечивается на уровне CUDA driver + kernel-mode module. MPS — это надстройка user-space (multi-process time-slicing на одном physical GPU context), она не меняет семантику cuMemcpy/cuLaunchKernel относительно ownership указателя. Оба режима отрабатывают одинаково.

Production-сервисы (uvicorn ×2, whisper, k2 FSA) продолжали работать в течение всего эксперимента без сбоев.

### 4. Гейт: 4 чужих production-сервиса — no impact

Тест был совершён при активной нагрузке ~20 GB VRAM от 4 сервисов uptime 5-8 дней. Тест-workload (4 KB alloc, 0.46 сек) не пересёкся ни с одним по VRAM, driver-scheduler справился с miksing без сбоев. **Смок-workload безопасен на гружёном GPU** — можно повторять на боевых машинах при желании.

### 5. Побочная находка операционного уровня — та же трещина `goml.Backend.Copy`

На sm_89 `goml.Copy(dst, foreign_host_ptr, byteLen)` тоже даёт `CUDA_ERROR_INVALID_VALUE (1)` (D2D-only path). Значит поведение backend-D2D-only не архитектурно-зависимое, а константное свойство goml API. Это gap для R03b integration.

## Обновлённая рекомендация фикса

R03a на sm_120 показал WORKS-WITH-CAVEATS. R03a-89 на sm_89 показал **то же самое поведение** до бит-эксактности:
- разные ctx handles численно на обеих;
- UVA cross-ctx memcpy и kernel launch — 100% успех на обеих;
- teardown безопасен на обеих;
- thread mix стабилен на обеих;
- инъекция ctx через public API невозможна на обеих (то же место `cuda..go:72`).

**Обновлённый вердикт по срочности фикса:**

- Ранее (R03a): «отложить, триггер — если UVA сломается на другой архитектуре».
- После R03a + R03a-89: двойной положительный сигнал по архитектурам.
- После R03a-120-mps: **четверной положительный сигнал** — полная матрица 2×2 `{sm_120, sm_89} × {MPS OFF, MPS ON}`. Не осталось конфигурации где UVA-cross-ctx мог бы «неожиданно сломаться» на нашем hardware.
- **Приоритет фикса A: LOW/DEFERRED окончательно.** Полезно как design-clean шаг, но не блокер интеграции — покрытие достаточно полное.

Единственный оставшийся неисследованный сценарий: **внешний CUDA-host в том же процессе** (PyTorch/JAX/TensorRT уже держит свой context, конфликт с goml не-primary). Это единственный случай, где путь B (context-injection в gotorch) может понадобиться. Проверить в R03b integration stage.

## Открытые вопросы для R03b

1. **goml.Backend.Copy — только D2D.** Гэп для H2D через интерфейс — константное свойство обеих архитектур. Требуется публичный `HostToDevice` в интерфейсе `Backend` для чистой интеграции.
2. **Refcount hygiene:** порядок Close (goml first vs gotorch first) не проверен — тесты в R03a-89 не покрывают.
3. **Внешний ctx-host** (PyTorch/JAX/TensorRT в том же процессе) — не проверен. Единственный оставшийся сценарий, где context-injection путь B может стать полезным.
4. ~~MPS-client mode~~ — **закрыт**. Прогон R03a-89 с `CUDA_MPS_PIPE_DIRECTORY=/tmp/nvidia-mps` показал 6/6 PASS bit-exact, идентично MPS OFF.

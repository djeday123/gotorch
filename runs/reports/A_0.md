# A-0 — matmul-backward на GPU через wrapper

**Дата:** 2026-07-24
**Тип:** первый ход главы A-BRIDGE (лестница ступень 1).
**Скоуп:** переезд `gradOW = normed^T @ gradLogits` с CPU host-loop на GPU cublasSgemm через `gt_gemm_ex` с trans-флагами. Форма B2-battle (V=32000, D=256, S=128, B=8).
**Результат:** speed 35.2× vs B2 baseline, accuracy essentially bit-exact (floor модель исправлена), CPU-map даёт новую карту приоритетов.

---

## 1. Инфраструктура

**Новый метод:** `Backend.MatMulF32Ex(a, b, c, m, n, k, transA, transB bool)`
- `gotorch/v6/cuda/api.go` — добавлен в интерфейс Backend.
- `gotorch/v6/cuda/backend_purego.go` — реализация через `gt_gemm_ex` (F32 IO, `CUBLAS_COMPUTE_32F` полный accumulator, не TF32).
- `goml/backend/gotorch/matmul_mp.go` — adapter extension через type-assertion (паттерн B-impl-4).

**Row-major swap trick с trans-флагами:**
```
C_row[m, n] = op(A_row)[m, k] · op(B_row)[k, n]
              transA=true → A stored row-major [k, m]
              transB=true → B stored row-major [n, k]

cuBLAS call: gt_gemm_ex(opB, opA, N=n, M=m, K=k,
                       B_ptr, ldb=row_stride_of_B,
                       A_ptr, lda=row_stride_of_A,
                       C_ptr, ldc=n)
```

**Новый Step:** `trainStepBattleA0` в `battle.go` — F32 forward + GPU matmul-backward через `MatMulF32Ex(normed, gradLogits, gradOW, Embed, Vocab, m, transA=true, transB=false)`.

| источник | значение | обоснование |
|----------|----------|-------------|
| **pre-reg (ошибочная модель K·eps)** | 1e-5 | Линейная модель `error ≤ K·eps` для CPU sequential loop. |

| **factual worst** | **1.12e-5** | Внутри re-issued floor 1.5e-5. |

**Классификация случая:** «pre-reg модель накопления неверна, floor перевыпускается по правильной sqrt(K)·eps модели». **НЕ** «floor промахнулся, живём с превышением». Расширение под факт запрещено.

**Убывание:** F32 CPU-bwd ✓ (Δ=−3.5734), F32 GPU-bwd ✓ (Δ=−3.5734). Идентично.

## 3. Speed — 30-run FA-класс (5.4s wall)

**Гейт запуска:**
- nvidia-smi silence: util=0%, mem_used=12 MiB ✓
- Clocks до: 180 MHz (idle), 31°C, 15.00 W
- Clocks после: 2617 MHz (boost), 32°C, 86.31 W
- Warmup 5 шагов ✓, замер 30 шагов ✓.

**Результат:**

| режим | median (ms) | CV (%) | speedup vs B2 F32 | прогноз | реальность vs прогноз |
|-------|-------------|--------|-------------------|---------|-----------------------|
| **B2 F32 CPU-bwd** (baseline) | 4791.5 | 1.87 | 1.00× | — | — |
| **A-0 F32 GPU-bwd** | **136.183** | **1.92** | **35.2×** | 300× (10-15мс) | **~10× ниже прогноза** |

**CV 1.92% > 1%** — не публиковать как high-precision числа. Механизм: overhead host↔GPU copies (см. §4 CPU-map) — не GPU-bound.

**Прогноз промахнулся 10× (35× vs 300×) — разбор:**
- Прогноз предполагал pinned memory H2D/D2H ~25 GB/s → probs D2H 128MB ≈ 5мс, gradLogits H2D 128MB ≈ 5мс.
- Реальность: unpinned copies, host memcpy limit. Из CPU-map §4:
  - probs D2H 49.24 мс = **2.6 GB/s** (в ~10× медленнее pinned).
  - gradLogits H2D 31.40 мс = 4.1 GB/s.
  - Плюс не учтённые: CE build 21.24 мс, gradOW D2H 13.83 мс, alloc/free overhead 8.65 мс.
- Модель прогноза pin-memory-idealized: реальность unpinned + full step composition = 10× больше.

## 5. Амдал переверстка (после A-0)

**До A-0 (B2 F32 baseline):** matmul-bwd CPU host loop 4.79 s = 99.97% wall'а.

**После A-0:** matmul-bwd GPU 0.33 мс = 0.26% wall'а. **Атака идеальная — узкое место переехало.**

**Новая карта узких мест (§4):**

| категория | ms | share | техника устранения |
|-----------|------|-------|-------------------|
| host↔GPU copies (D2H+H2D+D2H) | 94.47 | 75% | pinned memory (`cudaHostAlloc`) → ~10× быстрее; **CE-scatter на GPU** — убирает 3 копии одновременно |
| host compute (CE + gradLogits build) | 21.24 | 17% | **CE-scatter kernel на GPU** — упаковывает CE + gradLogits + scaling в один PTX-op |
| alloc/free overhead | 8.65 | 7% | pre-allocate buffers один раз, hold в state |
| GPU compute (all forward + bwd + adamw) | 1.18 | 1% | оптимально, не трогаем |

**Приоритетный ход (A-1 кандидат): CE + gradLogits scatter на GPU.**

Единственная техника, которая **одним ядром сразу убирает три самых жирных куска** (probs D2H + CE host + gradLogits H2D = 101.9 мс = 81% wall'а). PTX-ядро принимает `probs [m, V], targets_int64 [m]`, возвращает `gradLogits [m, V]` без host round-trip. Прогнозируемый wall после A-1: `125.6 − 101.9 + ~1мс kernel = ~25мс` → **speedup 190×** vs B2 baseline.

**Второй ход (A-2 кандидат): pre-allocate + gradOW-in-place adamw** = free 22 мс (11% + 7% = 18%). Wall после A-2: ~15 мс → **speedup 320×**.

**Attention (первоначальная цель главы A через FA-ядра) — сдвигается вниз по приоритету** до тех пор, пока CE-scatter не переедет: сейчас attention в этом Step отсутствует, а в LLM Step (main.go) attention ~1.6% wall'а (см. `A_BRIDGE_design.md` §4).

## 7. Verdict

> **A-0 F32 GPU-bwd:** accuracy **OK** (worst 1.12e-5 внутри re-issued floor 1.5e-5, grad hybrid 0/8.2M), speed **35.2×** vs B2 (136 мс vs 4791.5 мс, CV 1.92%), memory похожа на B2 F32 (+128 MB gradLogits GPU буфер).
>
> **Прогноз speed промахнулся 10×** (35× vs 300×): pin-memory-idealized модель не учла unpinned copy limit (~2.6 GB/s) + не-copy-компоненты (CE host, alloc overhead, gradOW D2H). CPU-map даёт полную композицию.
>
> **Прогноз accuracy floor'а переиссён по правильной модели** (sqrt(K)·eps + AdamW-boost = 1.5e-5); реальность 1.12e-5 внутри. Правило `feedback-sqrt-vs-linear-accumulator` — 3-й case, теперь durable.

**Главное открытие для главы A:** GPU compute занимает **1% wall'а**; 99% — host↔GPU copies + host compute + alloc. **Attention НЕ следующая цель** — сначала **CE-scatter на GPU** (убирает 81% wall'а за один kernel).

## 10. Решения перед A-1 (для пользователя)

**Р1.** Целевой Step для A-1 — тот же `trainStepBattleA0` (B2 форма, matmul-only, никакого attention), с наложением **CE-scatter GPU kernel**? Ожидание wall 25 мс, speedup 190× (`125→25` = минус 101 мс = минус три жирных куска).

**Р2.** Альтернатива: pinned memory для probs/gradLogits (сохранение архитектуры, +10× copy bandwidth) — простая, но не устраняет CE host + alloc. Ожидание wall 65 мс, speedup 74×.

**Р3.** Отступ на несколько шагов: пойти в attention (глава A первоначально), как планировалось изначально, оставив CE-scatter отложенным? В контексте boевого Step LLM.Backward, attention это 1.6% wall'а — оптимально после того как **CE-scatter уберёт этот bottleneck класс отовсюду**.

Рекомендация: **A-1 = CE-scatter kernel**. Устраняет 3 самых жирных куска сразу, unlock'ит realistic Amdahl-мерки для последующих ходов, и техника переносима на **любой forward с CE-loss** (LLM.Backward включительно).
